# Second Brain Cloud Architecture

## Overview
This repository uses a Google Cloud Run service (`secondbrain-gdrive`) to automatically sync, parse, and transcribe Viwoods `.note` files directly from Google Drive into `.md` Markdown files. 

## Cloud Infrastructure Details
- **Google Cloud Run:** The transcription logic (`cloud_function/main.py`) runs as a serverless Cloud Run instance.
- **Continuous Deployment (Cloud Build):** A Google Cloud Build trigger is hooked directly to this GitHub repository (`viwoods-brain2`). Every push to the `main` branch automatically builds a new container image and deploys it to the `secondbrain-gdrive` Cloud Run service in the `us-central1` region under the `aaron-secondbrain` GCP project.
- **GitHub Sync (Revision Control & Web App Backend):** After converting the `.note` files, the service pushes the generated `.md` files and any extracted handwriting images (to an `_attachments/` folder) to a specified private GitHub repository via the GitHub API (`PyGithub`). This repository acts as the direct backend for the web app frontend.
- **Authentication:** 
  - **In the Cloud:** The script uses Google's Application Default Credentials (`google.auth.default()`). 
  - **Locally:** If executed locally (e.g. via `utils/run_main.py`), it falls back to parsing a local `token.json` file.

## Critical Sync Logic & The "Infinite Loop" Fix
The system extracts high-resolution images from the proprietary `.note` zip archives and sends them to the Gemini 2.5 Flash API for OCR and transcription.

**The Timestamp Issue:**
We extract the true per-page modification timestamp from the internal `.note` JSON (`_NoteList.json` / `PageListFileInfo.json`) and back-date the generated Markdown file to match it. However, the user's tablet continuously syncs and overwrites the `.note` files, constantly bumping their Google Drive `modifiedTime` to the present.

**The Infinite Loop Resolution:**
To prevent the Cloud Run script from seeing `doc_time > md_time` and endlessly re-processing the same files in an infinite loop, the system implements a strict **MD5 Checksum Strategy**:
1. When generating a new Markdown file, the script captures the Google Drive `md5Checksum` of the raw source `.note` file.
2. It permanently bakes this checksum into the hidden `appProperties` (`{'source_md5': <hash>}`) of the Google Drive Markdown file during the `files().create()` or `files().update()` API call.
3. On subsequent runs, before even comparing timestamps, the script cross-references the source `.note` MD5 against the Markdown's saved `source_md5`. If they match, the file is instantly skipped, completely protecting the system from infinite loops caused by background tablet syncing.

## Web App Integration
- **GitHub Backend:** The frontend web app (`webapp/app.js`) securely fetches the file tree, Markdown notes, and image attachments directly from the private GitHub repository using the GitHub REST API and a Personal Access Token (PAT).
- **Google Tasks:** The app maintains isolated Google OAuth capabilities exclusively to retrieve and display your personal Google Tasks widget.
- **Markdown Processing:** The frontend intercepts the raw `> *Last updated: ...*` blockquotes generated by the Cloud Run function.
- **UI Styling:** It dynamically parses them via regex and transforms them into interactive, beautifully styled per-page timestamp badges (`<div class="page-timestamp">`) using styles defined in `webapp/styles.css`.

## Debugging and Tailing Logs
If you need to view the live Cloud Run output from any computer with the Google Cloud SDK authenticated, you can tail the logs directly via terminal:
```bash
gcloud beta run services logs tail secondbrain-gdrive --region us-central1 --project aaron-secondbrain
```
